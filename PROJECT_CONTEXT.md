# AI 漫剧一体化平台项目说明

## 1) 项目目标
- 目标是完成一个 AI 漫剧一体化平台：从文本创作（剧本）到视觉生成（图片）的完整流程。
- 当前阶段先完成核心后端与最小可用前端，保证端到端可跑通。

## 2) 你的核心要求（已对齐）
- 前后端统一使用 TypeScript。
- 剧本生成模型：
  - Base URL: `https://api.qnaigc.com/v1`
  - Model: `deepseek/deepseek-v3.2-251201`
- 图片生成模型：
  - Base URL: `https://api.qnaigc.com/v1`
  - Model: `kling-v2`
- 产品流程要求：
  - 先生成剧本，不自动生成图片。
  - 用户对剧本有最终决策权，可编辑。
  - 图片生成是可选动作，由用户手动触发。

## 3) 我在本项目中的角色
- 我是以“完成 AI 漫剧一体化平台”为目的的 AI 工程助手。
- 工作方式不是只写代码，还会结合产品思维推进：
  - 先定义用户流程与决策点。
  - 再落地工程实现与接口设计。
  - 最后执行自测与回归，保证功能可用。

## 4) 当前系统流程（已实现）
- Step 1: 前端提交 `prompt` 到 `POST /api/scripts/generate`，由 DeepSeek 生成 1~5 个分镜。
- Step 2: 用户在前端逐个分镜编辑。
- Step 3: 每个分镜可单独调用 `POST /api/images/generate` 生成图片。
- Step 4: 每个分镜在有图片后可单独调用 `POST /api/videos/generate` 生成视频（`kling-v2-6`）。
- Step 5: 在资源库统一查看图片与视频。
- Step 6: 可在前端将多个视频加入“待拼接序列”并调用后端做兼容性校验（暂不执行实际拼接）。
 - Step 6: 可在前端将多个视频加入“待拼接序列”，支持校验并一键自动合并。

## 5) 数据库与资源存储现状
- 当前数据库：**SQLite（本地）**，默认路径 `backend/data/app.db`。
- 当前图片存储位置：**SQLite BLOB**（后端拉取模型返回图片并入库）。
- 当前视频存储位置：**SQLite BLOB**（后端拉取模型返回视频并入库）。
- 访问方式：
  - 图片：`GET /api/images/:id`
  - 视频：`GET /api/videos/:id`
- 当前限制：
  - 单机本地存储，适合开发与小规模场景。
  - 暂未实现多机共享与对象存储归档。
  - 已接入轻量 `@ffmpeg-installer/ffmpeg`，支持后端自动合并。

## 6) 建议的下一步（产品+工程）
- 中期迁移到 PostgreSQL 保存：项目、剧本版本、生成任务、图片元数据。
- 接入对象存储（如 S3 / OSS / 七牛 Kodo）做图片转存与长期管理。
- 加入任务状态与重试机制，提升生成稳定性与可观察性。

## 7) 安全约定
- API Key 仅允许放在后端环境变量（如 `backend/.env`），不写入前端，不提交到仓库。
